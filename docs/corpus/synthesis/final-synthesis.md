The voice brief alone -- "write as if explaining to a curious peer, not selling to a skeptic" -- is better editorial guidance than most professional publications.

**2. Deep research on how to build it right.** 34 research files, 80+ sources, covering everything from Penn State brand hex codes to GitHub's agentic workflow guardrail model to the exact failure rate of AI-generated code (45% OWASP vulnerabilities). We know the pitfalls before we hit them.

What we don't have yet is a site that looks like it belongs next to smeal.psu.edu. That's what this synthesis is about.

## What Needs to Change (Evolve This)

### From Wix Template to Professional Site

The current site looks like what it is: a Wix free-tier site. The Wix banner at the top ("This website was built on Wix"), the wixsite.com URL, the slightly generic button styles, the limited layout control. The content is strong but the container diminishes it.

**The fix:** Rebuild in Next.js with Penn State brand tokens. Same content, professional container. The Nittany Navy + Pugh Blue palette on Roboto/Roboto Slab with proper whitespace immediately signals institutional affiliation. When someone from Smeal sees it, it should feel like it belongs in the Penn State ecosystem.

### From Static Pages to Structured Content

The Wix pages are opaque -- content locked in Wix's CMS, not versionable, not searchable, not maintainable by agents. The standalone HTML modules are better (we have the source) but they're iframe-embedded, not integrated.

**The fix:** All content in MDX with frontmatter metadata. The 10 module files become React components in `/content/smeal-majors/`. The Learn pages become MDX in `/content/learn/`. The news articles become MDX in `/content/news/`. Everything is version-controlled, diffable, and agent-editable.

### From Manual Updates to Agent-Maintained

Right now, updating the site means logging into Wix and clicking through a WYSIWYG editor. Only the person with the Wix account can make changes. When officers graduate, the site stagnates.

**The fix:** The entire site lives in a GitHub repo. Anyone with repo access and an AI coding agent can:

- Create an issue describing what they want to change
- Have an agent create a PR with the changes
- Get the PR reviewed and merged
- See the change deploy automatically via GitHub Pages

The agent reads AGENTS.md for context, follows the voice brief for content, respects Penn State branding via locked Tailwind tokens, and CI catches anything that doesn't meet standards.

### From Single-Maintainer to Club-Wide

The Wix site has one maintainer. The new site has:

- CODEOWNERS defining who reviews what
- Branch protection requiring review before merge
- CI that validates content, accessibility, and visual regression
- Agent instructions that any coding tool can read
- A doctor script for new contributor setup
- An issue-to-PR workflow where club members describe changes in English

### From Proof of Concept to Showcase

The current site proves the content is good. The new site proves the approach is good. An agent-maintained educational site at a top business school, built on open infrastructure, with a research corpus documenting every decision -- that's the kind of thing that gets attention from faculty, from other schools, and from the AI tooling ecosystem.

## The Path to Autonomous Maintenance

### Phase 1: Human-Built, Agent-Assisted (Now)

ajs10845 builds the site with agent assistance (Claude Code). Agent writes code, human reviews every change. The research corpus and voice brief guide decisions. CI catches regressions.

### Phase 2: Human-Directed, Agent-Executed (Next Semester)

Club members create GitHub Issues describing what they want ("add the April workshop event", "update the Finance module with new content about crypto regulation"). Agents pick up issues, create PRs, CI validates. A human reviewer approves and merges.

The `.claude/agents/` directory provides predefined task definitions for common operations. AGENTS.md tells agents how the project works. The voice brief tells them how content should sound.

### Phase 3: Agent-Maintained, Human-Supervised (Maturity)

Scheduled agentic workflows check for stale content, broken links, accessibility regressions, and dependency updates. Agents create PRs automatically. Humans review and merge on a cadence (weekly or bi-weekly).

The site is never abandoned because maintenance is automated. Officer turnover doesn't kill the site because the institutional knowledge is in the repo (corpus, ADRs, agent instructions, voice brief), not in anyone's head.

### What Makes This Work

1. **The content is the product, not the code.** Club members interact with MDX files and GitHub Issues, not React components.
2. **The guardrails are infrastructure, not instructions.** Branch protection, CI validation, Tailwind token locking, and accessibility testing catch problems mechanically.
3. **The taste is documented.** The voice brief, the four-section structure, the "field first, AI second" principle -- these are in the repo, readable by any agent, enforceable by any reviewer.
4. **The research corpus is the institutional memory.** When someone asks "why is it built this way?", the answer is in `docs/corpus/` and `docs/decisions/`.
